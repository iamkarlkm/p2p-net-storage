#基于固定长度数组和多级哈希的字符串存储系统

##设计思想：
1. 数据存储层：使用固定长度数组（文件），长度按 2 的幂次增长（4B, 8B, 16B... 128MB）。
   - 存储层只负责存储小于最大长度的数据,按向上取对应幂次块寻址bucket->内存映射文件存储.
   - 新增操作时必须传入数据类型(Index, Key, Value),实际数据,传入数据超过最大长度报错.
   - 返回数组下标作为数据询查询和更新操作的ID,操作时必须传入数据类型(Index, Key, Value),ID,实际数据长度(用于确定数据是在哪个固定长度的bucket)
   - 提供批量操作接口。新增时可以接受个二维数组,并根据实际的数据长度存入不同的bucket。并返回一个一维的ID数组,由调用方决定 ID归属。查询时如果传入一维的ID数组,必须指定数据尺寸以确定使用哪个bucket,也可以增加一个一维的长度数组确定使用对应的bucket。
2. 空间划分：分为共享空间（Shared）和独立空间（Independent）。每个空间包含 Index, Key, Value 三种格式。数据存储层应分别提供共享空间和独立空间的操作接口。
3. 存储逻辑：
   - Value: 根据长度存入独立空间对应Value Bucket 文件，返回 ValueID。
   - Key: 根据长度存入独立空间对应Key Bucket 文件，返回 KeyID。
   - Index: 组合 [KeyLen(4)+KeyHash32(4)+KeyID(8)+ValueLen(4)+ValueHash32(4)+ValueID(8)]，存入独立空间 Index Bucket 文件，返回 IndexID。
   - 哈希值运算统一使用DsDataUtil.java对应方法。
   - 使用8位哈希(256) -> 4bit位图(4*256=1024)+Index ID固定长度(32*8*256=65536),Hash Data总长度:1024+65536=66560bit,8320字节。由于不对应幂次块,为了不浪费空间,所以用独立的总控文件存储。
   - 4bit位图状态区分: 00 - 对应槽位空;01 - 对应槽位有Index ID;10 - 对应槽位有本文件64位数据块索引ID(offset=id值*8320),指向下一层哈希;11 - 当前层哈希冲突,需要扩展哈希位数,解决冲突,对应槽位有Index ID,但是由下一级哈希存储生成[Hash32->Hash64->Md5->Sha256];
   - Master: Hash32 -> IndexID，存入总控 Hash32 文件（DbHash32Index）。4个字节对应4层哈希:[byte0][byte1][byte2][byte3],最多4次寻址，查询和操作对应固定长度数据(当前是Index ID)
   - Master: Hash64 -> IndexID，存入总控 Hash64 文件（DbHash64Index）。8个字节对应8层哈希:[byte0][byte1][byte2][byte3]...,最多8次寻址，查询和操作对应固定长度数据(当前是Index ID)
   - Master: MD5 -> IndexID，存入总控 MD5 文件（DbMasterIndex）。16个字节对应16层哈希:[byte0][byte1][byte2][byte3]...,最多16次寻址，查询和操作对应固定长度数据(当前是Index ID)
   - Master: Sha256 -> IndexID，存入总控 Sha256 文件（DbSha256Index）。32个字节对应32层哈希:[byte0][byte1][byte2][byte3]...,最多32次寻址，查询和操作对应固定长度数据(当前是Index ID)
   - Key和Value对应不同系列的总控文件，互不干扰。
4. 查询和更新逻辑：
   - Key -> Value:[Hash32->Hash64->Md5->Sha256] -> IndexID (Master) -> Index Data (Index Bucket) -> Check Hash32/Key -> ValueID,Size -> [get/put Value Data (Value Bucket)].
   - Value -> Key:[Hash32->Hash64->Md5->Sha256] -> IndexID (Master) -> Index Data (Index Bucket) -> Check Hash32/Value -> KeyID,Size -> [get Key Data (Key Bucket)]。
5. 所有数据存储和查询,不在内存缓存任何数据(比如HashMap等),直接操作内存映射文件,理论上数据都在内存中,所以不考虑缓存。未来将考虑纹理内存映射文件的buffers。
